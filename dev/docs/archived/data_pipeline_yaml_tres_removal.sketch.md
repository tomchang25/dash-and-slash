# Remove Unused Entity YAML→Tres Pipeline

## Goal

Remove the generated entity YAML→tres pipeline and its supporting scaffolding, because no shipped game content ever adopted it and it is currently non-functional, while keeping the unrelated SFX YAML→wav/tres synthesis pipeline, which is real and in active use.

## Requirements

1. A fresh checkout no longer needs a generation step for any currently-shipped game content, because no shipped enemy, player, or other gameplay resource depends on generated `.tres` files — enemies and player stats are hand-authored on purpose (see `dev/docs/archived/enemy_data_backed_structure.md`, `enemy_data_resources.sketch.md`, `data_driven_player_stats.sketch.md`, each of which explicitly chose manual authoring over generated data).
2. The SFX synthesis pipeline (`data/yaml/sfx/*.yaml` → wav + `UiAudioEvent.tres`) keeps working unchanged, because it is a separate, real, in-use tool, not an instance of the entity pattern being removed.
3. The `/godot-test` snapshot workflow no longer runs the entity generation step, because that step currently fails (see Design) and blocks the whole snapshot on an unrelated, unused feature.
4. Project docs and standards that describe the generated-entity-registry pattern no longer instruct a contributor to run a broken/nonexistent workflow.
5. Illustrative code examples in unrelated standards docs stop referencing the removed `ExampleEntityData` class, because those are prose examples, not live consumers, and should keep compiling as read.

## Design

Two YAML pipelines currently exist in this repo and they are unrelated:

- **Entity pipeline** (`yaml_to_tres.py`, `tres_to_yaml.py`, `validate_yaml.py`, `yaml_stats.py`, `tres_lib/spec.py`, `tres_lib/registry.py`) — a generic "designer YAML → generated `.tres`" system inherited from the project's base template. It has exactly one registered entity spec, the `example_entity` template demo, and nothing in shipped gameplay code reads from it. `global/autoloads/registries/resource_registry.gd` (the base class this pattern's registries would extend) has zero subclasses in the codebase today. The pipeline is also currently broken: `tres_lib/registry.py` imports `tres_lib.entities.example_entity`, and that module does not exist in the repo, so `yaml_to_tres.py` raises `ModuleNotFoundError` immediately on run. `dev/tools/godot_test_setup.py`'s `_generate_tres()` step calls this same broken script and treats a non-zero return as fatal, so the `/godot-test` snapshot workflow is presently broken by this dependency too.
- **SFX pipeline** (`render_sfx.py`, `sfx_synth.py`, `data/yaml/sfx/*.yaml`) — deterministic procedural audio synthesis (seed → WAV + `UiAudioEvent.tres`). It shares two low-level, generic `.tres`-writing helpers with the entity pipeline (`tres_lib/tres_writer.py`, `tres_lib/uid.py`) but does not import `tres_lib/spec.py` or `tres_lib/registry.py`. It is unaffected by removing the entity pipeline as long as the shared helpers stay.

Enemies and player stats each have their own archived design docs stating, as an explicit requirement, that generated data was rejected because the entity count is small and hand-authored resources plus per-class code handle the needed diversity better. This sketch does not re-litigate that decision — it just removes the now-dead alternative path so the codebase stops carrying a broken, unused system.

## Sketch (non-normative)

Remove:

- `dev/tools/yaml_to_tres.py`
- `dev/tools/tres_to_yaml.py`
- `dev/tools/validate_yaml.py`
- `dev/tools/yaml_stats.py`
- `dev/tools/tres_lib/registry.py`
- `dev/tools/tres_lib/spec.py`
- `dev/tools/tres_lib/tres_format.py` (spot-checked: not imported anywhere, including by `tres_writer.py`; implementer confirms before deleting)
- `dev/tools/prompts/yaml_generation/` (entity-creation prompt pack built around this pipeline, per the 2026-06-25 CHANGELOG "Example" entry)
- `data/yaml/example_entity.yaml`
- `data/definitions/example_entity_data.gd` (+ `.gd.uid`)
- `global/autoloads/registries/resource_registry.gd` (+ `.gd.uid`)
- `global/autoloads/resource_dir_loader.gd` (+ `.gd.uid`) — only consumer is `resource_registry.gd`
- `dev/foundation/platforms/godot/standards/registries.md` — no generated-registry type exists in the codebase to document; recreate this standard fresh if/when a real one ships, rather than keeping a standard with nothing to point at

Keep unchanged:

- `dev/tools/render_sfx.py`, `dev/tools/sfx_synth.py`
- `dev/tools/tres_lib/tres_writer.py`, `dev/tools/tres_lib/uid.py`
- `data/yaml/sfx/**`

Update:

- `dev/tools/godot_test_setup.py` — drop `_require_yaml()` / `_generate_tres()` and their call sites in `main()`, since there is no entity generation step left to run before a test snapshot.
- `global/constants/data_paths.gd` — drop `EXAMPLES_DIR` and the "generated by the YAML→tres pipeline" docstring warning; nothing else currently reads a generated directory through `DataPaths`.
- `CLAUDE.md` root file and `dev/agent_rules/agent_startup.md` — the `## Data Pipeline` section currently says "Entities are authored in `data/yaml/*.yaml`, converted to `.tres` via `dev/tools/yaml_to_tres.py`..."; replace it with a short section describing only the SFX synthesis workflow (`render_sfx.py` reads `data/yaml/sfx/*.yaml`, writes wav + `UiAudioEvent.tres`, never hand-edit the outputs).
- `dev/standards/project_structure.md` — remove the `data/tres/` generated-directory line, the "Generated by `dev/tools/yaml_to_tres.py`" line, the `registries/` per-type-registry-autoloads line, and the data-pipeline/registry-autoload table rows; keep `data/yaml/sfx/` as the one real YAML source.
- `dev/foundation/platforms/godot/standards/naming_conventions.md`, `dev/foundation/platforms/godot/standards/gdscript_structure_standard.md`, `dev/foundation/platforms/godot/skills/godot4_theme_override.md` — swap the illustrative `ExampleEntityData` references for a different placeholder type name (or an existing real type) so the example code doesn't cite a deleted class.

Migration steps:

1. Grep for `ExampleEntityData`, `EXAMPLES_DIR`, `ResourceRegistry`, `ResourceDirLoader`, `yaml_to_tres`, `tres_to_yaml`, `validate_yaml`, `yaml_stats` across the repo one more time immediately before deleting, to catch anything not listed above.
2. Delete the entity-pipeline scripts, `tres_lib` modules, and example scaffolding listed above.
3. Update `godot_test_setup.py`, `DataPaths`, `CLAUDE.md`/`agent_startup.md`, `project_structure.md`, and the three standards docs with illustrative-only references.
4. Run `dev/tools/lint_standards.py --files <changed>` on everything touched.
5. Run `/godot-test` (or the equivalent snapshot setup) end to end to confirm the snapshot step no longer depends on the removed pipeline.
6. Run `dev/tools/render_sfx.py --dir data/yaml/sfx/ --godot-root .` (or the project's normal SFX regen path) to confirm the SFX pipeline still renders after the shared `tres_lib` trim.

## Non-Goals

1. Do not touch the SFX YAML pipeline or its output files.
2. Do not touch enemy or player hand-authored `.tres` resources or their `.gd` data classes — they already ship as intended and are out of scope.
3. Do not design or build a replacement generated-data pipeline as part of this work.
4. Do not resolve the open TODO drafts about data-driven enemy spawn weights or player attack range as part of this work; those stay independent future decisions, not blocked by this removal.

## Acceptance Criteria

1. A fresh checkout and a `/godot-test` snapshot run complete without invoking any entity YAML→tres generation step.
2. The SFX pipeline still renders `data/yaml/sfx/*.yaml` into wav + `.tres` output unchanged.
3. No project doc instructs a contributor to run the removed entity pipeline or points at `dev/foundation/platforms/godot/standards/registries.md`.
4. No reference to `ExampleEntityData`, `EXAMPLES_DIR`, `ResourceRegistry`, or `ResourceDirLoader` remains anywhere in the codebase.
