**Tick arena runtime ownership split**

Status: Resolved probe.

Decision: reviewed on Opus. Facets #3 (TickActionController presentation split) and #4 (preview aim/plan dedup) handed to `tick_arena_combat_feedback_and_aim.implementation_spec.md`. Facets #1 (relocate pure combat files to `common/`) and #2 (extract a TickEngine contract) rejected as premature and closed. Probe archived.

Resolution — the four open questions below are answered as follows:

- #1 (move the five already-pure combat files to `common/gameplay/combat/`) → **do not move.** "Pure" (no scene coupling) is a mechanical property, not an ownership one. `TickCombatRules`, `TickCombatProjection`, `TickHitResolver`, `TickHitOutcome`, and `TickActionPlanner` encode tick-arena policy and tuning (they are even `Tick*`-named), not reusable primitives like `TileDirectionResolver`/`Health`/`Guard` that already earned their place in `common/`. With exactly one gameplay feature consuming `common/`, promoting policy there is speculative generality and would make the class names lie. They stay in `game/tick_arena/combat/`.
- #2 (extract a TickEngine actor/board contract) → **do not extract now.** An interface earns its keep only when a second board/actor implementation needs it; none exists. The `%GridArena`/`%Player`/`GridEnemy` coupling is honest under a single scene. Revisit only when a second consumer is real.
- #3 and #4 → **actionable, spec written.** See `tick_arena_combat_feedback_and_aim.implementation_spec.md`.

The felt "common/tick_arena overlap" is diagnosed as one god-object (TickActionController) plus a boundary that is under-defined because there is a single consumer — the boundary will define itself when a second gameplay feature appears. Original survey retained below as the record behind these decisions.

`entity_layer_legacy_mixing.probe.md` covered `game/entities/` and the `common/gameplay/components/` physics-hitbox legacy; that residue is resolved. The overlap this probe covers is different: `common/gameplay/` is now a clean board/combat/entity/view contract layer (`GridArena` as terrain/occupancy/reservation authority, `Health`/`Guard`/`TileDirectionResolver` as combat state/rules, `EnemyStatusBars`/`ShieldStatusIcon` as view), but `game/tick_arena/` still carries most of its domain logic directly inside scene controllers instead of composing over shared rule/contract objects. This probe surveys `game/tick_arena/` class by class so the next split decision has evidence instead of impression.

- **Already pure, not yet moved** — `tick_action_planner.gd` (no `extends`, static geometry only), `tick_combat_rules.gd` (no `extends`, static numbers/angle naming), `tick_combat_projection.gd` (`extends RefCounted`), `tick_hit_resolver.gd` (`extends RefCounted`), and `tick_hit_outcome.gd` (`extends RefCounted`, typed value object) all live in `game/tick_arena/combat/` but take no scene-tree references — every dependency arrives as an explicit argument (`GridArena`, `TickEngine`, `RunBuild`, resolved cells/values). These match the "pure rule/typed contract" bucket the prior discussion flagged as low-risk to relocate into `common/gameplay/combat/`. `d556cbb` already did the equivalent move for `EnemyHitResolver` → `TickHitResolver` routing, so this bucket has precedent.

- **Scene-bound domain owner, not common-ready** — `tick_engine.gd` (`extends Node`) hard-binds `@onready var _grid: GridArena = %GridArena` and `@onready var _player: TickPlayer = %Player`, and every public method signature is typed to `GridEnemy`/`TickPlayer` directly (`register_actor(actor: GridEnemy)`, `damage_player(amount, source: Node)`, `is_cell_open_for_player`). It owns the tick counter, the actor energy scheduler, and world-advance resolution order — genuine domain authority — but the coupling to `%GridArena`/`%Player` unique-name lookups means it cannot move to `common/` without first extracting an actor/board contract interface.

- **One controller, five responsibilities** — `tick_action_controller.gd` (512 lines) is the only file that still bundles distinct concerns end to end: verb legality/dispatch (`handle_verb`, `_verb_move/_attack/_dash/_smash/_wait`), player action economy (`_spend_speed_if_full`, `_fill_speed_meter`, `_tick_player_action_upkeep`, `_mobility_action_refunds`), hit application (`_apply_player_hit` calling `enemy.take_hit(...)`), Major-trigger VFX/SFX playback (`_play_major_trigger_feedback` calling `MajorTriggerFeedbackVFX`/`AudioManager` directly), and HUD message text construction (`_apply_player_result_message`, `set_message`, `_append_message_suffix`). Its own header comment already states the ownership boundary in prose ("may mutate TickPlayer, RunBuild, enemy health... and TickGridView feedback flashes; may request world advancement from TickEngine, but never owns tick count, actor energy, wave state, reward state, spawn queues, or terrain cadence") — the header is accurate, but the boundary is asserted in a comment rather than enforced by file structure.

- **Preview controller duplicates the action controller's aim/plan wrappers** — `tick_preview_controller.gd`'s `_mouse_cell()`, `_aim_direction()`, `_compute_dash_plan()`, `_clamped_smash_target()`, and `_chebyshev()` are structurally identical wrappers around `TickActionPlanner` calls to the ones in `tick_action_controller.gd`, differing only in reading `action_controller.get_last_aim()` instead of `_last_aim` directly. This is a reuse gap adjacent to the ownership split, not a legacy-residue issue — both controllers already depend on `TickActionPlanner`, so the wrapper duplication could collapse into one shared aim-context helper the same pass touches `tick_action_controller.gd`.

- **Reward/wave/run/view groups are internally consistent, not part of this split** — `game/tick_arena/reward/effects/*` (15 files) already follow one uniform `WaveRewardEffectDefinition`/`MajorEffect` inheritance shape; `wave/` (`wave_controller.gd`, `enemy_spawner.gd`, `enemy_spawn_planner.gd`, `wave_scaling.gd`) are `RefCounted` scene-local coordinators with no cross-cutting overlap into `combat/`; `run_build.gd` is pure `RefCounted` data the prior discussion already flagged as "awkward, don't move yet, wait for reward/run ownership to be decided together." None of these show the same common/tick_arena straddling the `combat/` folder does, so they stay out of scope for this split unless the survey below finds otherwise.

The discussion target: confirm the three-tier split — (1) relocate the five already-pure `combat/` files into `common/gameplay/combat/` now, since they have no scene coupling to strip; (2) decide whether `TickEngine` gets an extracted actor/board contract before or independent of the `TickActionController` split, since `TickActionController`, `TickPreviewController`, and every `GridEnemy` all depend on its current concrete API; (3) decide the actual seam for splitting `TickActionController`'s five responsibilities — the header comment already names four of them (verb dispatch, economy, hit application, feedback), so the seam may already be implied and just needs to be turned into files.

Open question: does relocating `tick_combat_rules.gd`, `tick_hit_resolver.gd`, `tick_combat_projection.gd`, `tick_hit_outcome.gd`, and `tick_action_planner.gd` into `common/gameplay/combat/` happen as its own small first PR, or does it wait and land together with the `TickActionController` split so the moved files' new call sites are updated only once?

Open question: should `TickEngine`'s actor/board contract extraction happen before `TickActionController` is split, so the new action-resolver/economy files are written against the final contract instead of against `%GridArena`/`GridEnemy` directly and then updated again later?

Open question: for the `TickActionController` split, is the intended shape verb-orchestrator + action-resolver/applier + action-economy-helper + feedback(VFX/SFX/HUD)-extraction exactly four files, or does economy stay inlined in the orchestrator since it is only ~30 lines total (`_spend_speed_if_full`, `_fill_speed_meter`, `_tick_player_action_upkeep`, `_mobility_action_refunds`)?

Open question: does the `TickPreviewController` wrapper duplication get folded into the same pass (e.g. both controllers taking a shared `TickAimContext` instead of each wrapping `TickActionPlanner` separately), or is it tracked as a separate small cleanup so it doesn't inflate the ownership-split scope?
