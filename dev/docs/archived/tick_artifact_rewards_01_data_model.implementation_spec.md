# Tick Artifact Rewards 01: Data Model

Parent Plan: `tick_artifact_rewards.md`

## Goal

Replace the tier-split reward-effect class hierarchy with one composed `Artifact` type that carries identity plus a list of effect contributions, turning rarity, stack rule, exclusivity, and curse status into data instead of subclasses. This is the foundation the roll (child 02) and inspection (child 03) work reads. Behavior is unchanged; only the data shape and the run-scoped registry change.

## Summary

Today a reward effect is a subclass of `WaveRewardEffectDefinition`: one class per channel for Minors, plus a `MajorEffect` intermediate with one subclass each for Smash, Guard Shredder, Execution, and Flowing Strike. Adding or inspecting a reward means reasoning about a class tree. This child collapses that tree into one `Artifact` value object whose behavior is a small list of `ArtifactEffect` contributions â€” a channel amount, a mobility payload replacement, or a mobility trigger activation. The four carried-over Majors become artifacts whose effect list holds a payload or trigger op; their applied result against `RunBuild` is bit-for-bit what it is today (acceptance criterion 2).

This is the **Choice A** scope: the point-balancing roll stays working. `WaveRewardChoiceGenerator` keeps its profile/target-point/rejection-sampling machinery and its code-constructed pool; only the pool's element type changes from concrete effect subclasses to `Artifact` instances, and its one eligibility call switches to the unified predicate. Deleting the generator is child 02's job, not this one. Because the deleted base class currently hosts the `Profile` enum that the generator and `WaveRewardChoice` still need, `Profile` relocates to `WaveRewardChoiceGenerator`; the `Tier` enum is deleted outright since nothing in the roll branches on it.

`RunBuild`'s Major bookkeeping (`_major_entries`, `MAJOR_CAP`, `add_major`/`has_major`/`can_add_major`) generalizes into one owned-artifact registry of `{ artifact, stacks }` entries, with the run-wide cap re-keyed from "count of Majors" to "count of legendary-rarity artifacts" (same numeric value, 4). Every picked artifact â€” Minor or Major â€” now registers here, which is what makes the single registry and single eligibility predicate possible and what child 03's inspection panel will read. The channel-total, payload-override, and trigger-set portions of `RunBuild` are untouched, so every downstream reader (Player, TickPlayer, TickArena, WaveController, the combat controllers) keeps consuming the exact same API.

The reward context drops its unused legacy-player parameter â€” verified `null` at its one production construction site and every test site â€” narrowing to `(grid, run_build)`. The tiny `WaveRewardApplier` (a three-line loop, one call site) is inlined into `WaveRewardChoiceController` and deleted. The dead `PlayerStatEffect` base (its `is_applicable` hard-returns `false`) is deleted, and its lone surviving subclass Attack Range becomes an ordinary always-eligible channel artifact.

Result: one artifact type with a composed effect list replaces roughly fifteen effect files; carried-over behaviors apply identically; the registry and eligibility predicate are unified; lint and unit tests pass. Rarity is authored on every artifact but, beyond gating the legendary cap, stays dormant until child 02 consumes it for weighted rolling and child 03/overlay for card color.

## Relational Context

- `WaveRewardChoiceGenerator` owns the pool and the roll; it reads each `Artifact`'s `allowed_profiles`, `min_wave`, `point_value`, `max_stacks`, and `is_eligible(context)` to build point-balanced offers. This call direction and the point machinery are preserved verbatim under Choice A.
- `RunBuild` is the single authority for run-scoped applied state: channel totals, mobility payload override, mobility trigger set, and now the owned-artifact registry. Eligibility predicates and appliers **read and write** it; they never hold a second copy of that state.
- The unified eligibility predicate lives on `Artifact.is_eligible(context)` and mirrors today's `MajorEffect.is_applicable`: it gates on `min_wave`, rejects an already-owned unique (`max_stacks == 1`) artifact via `RunBuild.has_artifact(id)`, and rejects an exclusivity conflict or a full legendary slot via `RunBuild` query methods. `RunBuild.acquire_artifact` stays authoritative on write and returns `false` (with a dev-error) on a rejected add, exactly as `add_major` does today â€” the predicate is the pre-offer filter, not the guard of record.
- Applying one rolled artifact does two things in order: register it in the owned registry with its stack count, then apply each `ArtifactEffect` against `RunBuild`. A stackable pick increments the existing entry's stacks; a unique re-pick is rejected at registration.
- `ArtifactEffect.apply` writes only through `RunBuild`'s existing mutators (`record`, `set_mobility_payload_override`, `set_mobility_trigger`) â€” the same calls the current effect subclasses make. No new `RunBuild` write surface for channels/payload/triggers.
- The combat and player readers (`tick_action_controller.gd`, `tick_preview_controller.gd`, `tick_arena.gd`, `wave_controller.gd`, `tick_player.gd`) consume `RunBuild.total`, `get_mobility_payload`, and `has_mobility_trigger` only. They are outside this change's blast radius and must not be touched.
- `WaveRewardChoice` and `WaveRewardOverlay` keep their current shape (profile, target points, description lines); the only edit they need is the `WaveRewardEffectDefinition.Profile` â†’ `WaveRewardChoiceGenerator.Profile` reference move. The overlay is otherwise untouched.
- The legendary cap is a rename of the Major cap's semantics: `MAJOR_CAP` (4) becomes a legendary-count cap (4). The four carried-over Majors are authored as `LEGENDARY` rarity so the cap gates exactly the set it gates today.

## Scope

### Included

- New `Artifact` value type and `ArtifactEffect` abstract with `ChannelArtifactEffect`, `PayloadArtifactEffect`, `TriggerArtifactEffect` concretes.
- Re-authoring the generator's code-constructed pool as `Artifact` instances (Minors, the four Majors, and Attack Range).
- Generalizing `RunBuild`'s Major store into one owned-artifact registry with a legendary-rarity cap and unified query methods.
- Unified `Artifact.is_eligible(context)` predicate; the owned wrapper (`WaveRewardEffect`) rebinds from `definition` to `artifact` and registers on apply.
- Dropping the legacy-player parameter from `WaveRewardContext`; inlining and deleting `WaveRewardApplier`.
- Relocating the `Profile` enum to `WaveRewardChoiceGenerator`; deleting the `Tier` enum, the effect subclasses, `MajorEffect`, `MajorPlaceholderEffect`, and `PlayerStatEffect`.
- Updating the affected unit tests to the artifact shape.

### Excluded

- Any change to the roll algorithm, cadence, curse-pool wiring, or profile/point removal â€” child 02.
- Inspection UI â€” child 03.
- Moving artifact authoring to `.tres` resources â€” later follow-on; pool stays code-constructed.
- Rarity-weighted rolling or card-color rendering â€” rarity is authored but dormant here.
- Any change to `RunBuild`'s channel/payload/trigger read API or its consumers.

## Implementation Notes

- **Roll metadata stays on `Artifact` under Choice A.** `point_value`, `magnitude`, and `allowed_profiles` survive on the artifact because the generator still balances by points and profiles. They are removed only when the generator is deleted in child 02. Do not strip them here chasing the sketch's final shape.
- **`magnitude` is the display number; the channel effect owns the applied amount.** For a single-channel artifact these are the same value, authored to match, so `description()` (which formats `magnitude * stacks` into `description_template`) reads identically to today. `ChannelArtifactEffect` carries its own `amount` plus a `unit_scale` (1.0 for flat channels, 0.01 for the percent-authored pressure channels that currently divide by 100). Keep the enemy-pressure percent scale exactly as the current subclasses apply it.
- **Every pick registers now.** Applying any artifact calls `RunBuild.acquire_artifact(artifact, stacks)` before applying its effects. This is new for Minors (previously only Majors registered) but adds no gameplay output change â€” channels still receive identical deltas. It is what makes one registry and one predicate possible.
- **Eligibility mirrors the old Major contract.** `Artifact.is_eligible(context)` returns `min_wave` satisfied AND (`max_stacks > 1` OR not `has_artifact(id)`) AND no exclusivity conflict AND, for legendary rarity, a free legendary slot. `acquire_artifact` re-checks these on write and dev-errors on rejection â€” never trust the caller's pre-filter alone.
- **Legendary cap = renamed Major cap.** The numeric cap stays 4; it now counts `LEGENDARY`-rarity owned artifacts. Author Smash, Guard Shredder, Execution, and Flowing Strike as `LEGENDARY`; author all Minors and Attack Range as `COMMON`. Smash keeps its `mobility_slot_replacement` exclusivity group; the other three keep an empty group.
- **`Tier` is deleted, not migrated.** Nothing in the roll branches on tier; legendary-vs-not is derived from `rarity`. Confirm no reader of `is_major`/`is_minor`/`Tier` remains after deletion (the only ones were on the deleted base class).
- **Test parity.** `test_major_effect.gd`'s cap/exclusivity/unique cases map directly onto `acquire_artifact` with authored legendary artifacts; the Smash/Guard/Execution/Flowing-Strike tests assert the same `get_mobility_payload`/`has_mobility_trigger` outcomes after applying an authored payload/trigger artifact; the channel tests author channel artifacts and assert the same `total(channel)` results.

## Edge Cases

| Case                                                                        | Expected Handling                                                                                                                                                              |
| --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Stackable Minor picked across multiple waves                                | `acquire_artifact` increments the existing entry's stacks; no cumulative cap is enforced here (per-offer roll cap remains the generator's concern), matching current behavior. |
| Unique (legendary) artifact re-offered                                      | `is_eligible` returns `false` via `has_artifact(id)`; a forced `acquire_artifact` is rejected with a dev-error.                                                                |
| Legendary slot full at a milestone                                          | `is_eligible` returns `false` for further legendaries; the generator simply rolls from the remaining eligible pool (fallback shaping is child 02).                             |
| Attack Range (formerly `PlayerStatEffect`, always `is_applicable == false`) | Becomes an ordinary always-eligible `COMMON` channel artifact on the attack-range channel; it now participates in offers instead of being permanently filtered out.            |
| `RunBuild.clear()` on run reset                                             | Clears the owned-artifact registry alongside channels, payload override, and triggers, exactly as it clears `_major_entries` today.                                            |

## Acceptance Criteria

1. One artifact type with a composed effect list replaces the tier hierarchy and every concrete effect subclass; the point-balancing roll still produces offers.
2. Smash, Guard Shredder, Execution, and Flowing Strike apply identically to today, now as authored artifacts, verified by their existing observable outcomes.
3. The reward context carries no legacy player, and no code references one.
4. The run-scoped store exposes one owned-artifact registry with a rarity-keyed legendary cap and one eligibility predicate; every picked artifact registers there.
5. Lint passes on all changed files and the unit suite passes.
