# Data-Driven Wave Progression And Enemy Levels 01: Progression Data Model

Parent Plan: `data_driven_wave_progression_and_enemy_levels.md`

## Goal

Explore the authored data boundary for ordered waves, conditional groups, enemy compositions, level offsets, and the shared four-stat enemy-level curve before runtime migration begins.

## Summary

The favored direction is a wave-domain resource graph rather than more constants in the current formula helper. One catalog should expose ten explicit demo waves and one reusable endless template. Each wave should own ordered group resources, while one progression profile owns the level-to-stat math shared by all enemy kinds.

Enemy scenes and attack profiles currently contain much of the Level 1 tuning, while runtime wave scaling mutates health and stores damage and Defense adjustments. The later implementation spec should verify whether consolidating all four base stats into enemy-owned tuning resources is required for consistent projection, especially because Guard currently lives on a scene component and attack damage lives on attack data.

## Sketch

- Candidate resource roles are a wave catalog, wave definition, ordered group definition, enemy composition entry, spawn condition, and enemy-level progression profile. Prefer nested hand-authored resources where reuse does not improve designer clarity; the catalog should remain easy to audit as the complete progression surface.
- The wave catalog likely belongs to a new wave data domain or alongside the owning tick-arena wave feature, following the project's rule that hand-authored gameplay tuning resources live with their feature. The later spec must verify placement against existing enemy-data conventions.
- A wave definition should likely carry display identity, ordered groups, and the population safety cap. Milestone or elite booleans should not remain authoritative once the group data can directly identify special entries and completion behavior.
- A group definition should likely carry composition entries, total count or explicit counts, start-condition kind, predecessor survivor threshold, warning duration if it differs from the shared default, and level offset. The runtime must not infer special enemy behavior from scene identity.
- Composition entries should reference enemy content and an authored weight or count. Explicit counts are favored for the ten demo waves and boss group; weighted selection may be useful inside a fixed support group but should remain deterministic under a seeded run.
- The endless template should be a first-class authored definition rather than generated by copying the wave-10 object at runtime. That keeps its fixed grammar reviewable and lets the boss encounter remain unique to the official demo finale.
- The progression profile should likely expose a base, post-demo, and lethal segment for each stat, with coefficients and exponents that feed the parent formula. It should also own rounding rules, minimums, and any safe numerical bounds rather than scattering them across enemy consumers.
- Level 1 base HP, Guard, Defense, and attack damage need one unambiguous ownership path per enemy kind. Verify the current split between Health nodes, Guard nodes, attack resources, and runtime Defense before choosing whether enemy data expands or a separate combat-stat profile is cleaner.
- All attack variants belonging to one enemy should receive the same level-derived damage multiplier so authored differences between attacks remain intact. Avoid rewriting every attack's base damage during spawn.
- Guard projection must set both maximum and current Guard before the enemy becomes active. Health projection must similarly produce full scaled health at spawn. Reprojection during an enemy's lifetime is out of scope.
- Existing reward pressure channels should consume the already projected stats as explicit post-level modifiers. Their presentation should remain separate from displayed level so a Level 12 enemy has stable base expectations even when a run curse makes it stronger.
- Candidate files to inspect: `game/tick_arena/wave/wave_scaling.gd`, `game/tick_arena/wave/wave_controller.gd`, `game/entities/enemies/grid_enemy.gd`, `game/entities/enemies/data/`, `data/enemies/definitions/`, enemy scenes containing Health and Guard tuning, reward-pressure resources, and wave/enemy scaling tests.

## Non-Goals

1. No runtime ordered-group scheduler or migration of active wave flow.
2. No demo-completion UI or endless transition.
3. No final authored wave counts, compositions, or curve constants beyond fixtures needed to validate the model.
4. No new enemy type or final boss data.

## Acceptance Criteria

1. The complete ten-wave demo and one fixed endless template can be represented without special-case runtime flags for elite or boss timing.
2. Every group expresses composition, order, predecessor-survivor condition, population-limited warning behavior, and level offset in authored data.
3. One progression profile deterministically maps displayed level to HP, damage, Guard, and Defense projections across the demo, endless, and lethal regions.
4. Level 1 enemy identity remains controlled by enemy-owned base tuning, and reward pressure remains distinguishable from level growth.
